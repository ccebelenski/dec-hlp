# FFI Bindings Design: C/C++ and Python

Version: 1.0-draft
Date: 2026-03-10

This document specifies the design of two binding crates that expose the core `dec-hlp` Rust library to C/C++ and Python consumers. It also defines the Cargo workspace layout that ties all crates together.

### Why the CLI Is a Separate Crate

The CLI binary (`hlp`) is extracted into its own workspace member for three reasons:

1. **Dependency isolation.** The CLI depends on `clap`, `crossterm`, and other user-facing crates. Keeping these out of the core library crate means `dec-hlp-ffi` and `dec-hlp-python` do not transitively pull them in.
2. **Build artifact clarity.** `cargo build -p hlp` produces only the binary. `cargo build -p dec-hlp-ffi` produces only the shared/static libraries. No ambiguity about which `lib` or `bin` target is intended.
3. **Independent versioning potential.** The CLI and library can be versioned independently if their release cadences diverge, though initially they share a version.

## 2. `dec-hlp-ffi` Crate -- C ABI Wrapper

### 2.1 Crate Configuration

```toml
# dec-hlp-ffi/Cargo.toml
[package]
name = "dec-hlp-ffi"
version = "0.1.0"
edition.workspace = true
rust-version.workspace = true
license.workspace = true
description = "C-compatible FFI bindings for the dec-hlp VMS help library"
publish = false  # Not published to crates.io; distributed as .so/.a + header

[lib]
crate-type = ["cdylib", "staticlib"]
# cdylib  -> libdechlp.so  (or libdechlp.dylib on macOS)
# staticlib -> libdechlp.a

[dependencies]
dec-hlp = { workspace = true }

[build-dependencies]
cbindgen = "0.28"
```

The dual `crate-type` produces both `libdechlp.so` (for dynamic linking) and `libdechlp.a` (for static linking) from a single build.

### 2.2 Design Principles

**Opaque handles.** All Rust types are hidden behind opaque pointers. C callers never see Rust struct layouts, never allocate Rust objects, and never dereference into Rust memory. This insulates the C ABI from Rust struct layout changes.

**Integer return codes.** All functions return `i32` status codes. Zero means success; negative values indicate errors. The specific error message is retrievable via a thread-local last-error function.

**Thread-local last-error.** Following the `errno` / `GetLastError()` pattern, each FFI function that can fail stores a detailed error message in a thread-local buffer. The caller retrieves it with `dechlp_last_error()`. This avoids passing error string pointers across the FFI boundary as out-parameters.

**Null-terminated C strings.** All string parameters are `const char *` (null-terminated UTF-8). All string return values are either caller-owned (must be freed) or borrowed (valid until the next API call on the same handle).

**Explicit ownership.** Every allocation has a matching free function. The header documents ownership for every pointer parameter and return value.

### 2.3 Return Code Convention

```c
/* Return codes */
#define DECHLP_OK             0    /* Success */
#define DECHLP_ERR_NULL_ARG  -1    /* A required pointer argument was NULL */
#define DECHLP_ERR_IO        -2    /* I/O error (file not found, permission, etc.) */
#define DECHLP_ERR_FORMAT    -3    /* Invalid .hlib file format */
#define DECHLP_ERR_NOT_FOUND -4    /* Topic not found */
#define DECHLP_ERR_AMBIGUOUS -5    /* Topic name is ambiguous */
#define DECHLP_ERR_BUILD     -6    /* Build failed (parse error or empty tree) */
#define DECHLP_ERR_INVALID   -7    /* Invalid argument (e.g., out-of-range index) */
#define DECHLP_ERR_INTERNAL  -99   /* Unexpected internal error (Rust panic caught) */
```

### 2.4 C Header: `dec_hlp.h`

The header is generated by `cbindgen` from annotated Rust source and checked into the repository so C consumers do not need Rust tooling.

```c
/*
 * dec_hlp.h -- C bindings for the dec-hlp VMS help library
 *
 * This file is generated by cbindgen. Do not edit manually.
 *
 * Memory ownership rules:
 *   - Handles returned by dechlp_*_open/create must be freed with the
 *     corresponding dechlp_*_close/destroy function.
 *   - String pointers returned by functions are valid only until the next
 *     call on the same handle, unless documented otherwise.
 *   - dechlp_string_free() frees strings returned by functions that
 *     transfer ownership (documented as "caller must free").
 */

#ifndef DEC_HLP_H
#define DEC_HLP_H

#include <stddef.h>
#include <stdint.h>

#ifdef __cplusplus
extern "C" {
#endif

/* --- Return codes --- */

#define DECHLP_OK             0
#define DECHLP_ERR_NULL_ARG  -1
#define DECHLP_ERR_IO        -2
#define DECHLP_ERR_FORMAT    -3
#define DECHLP_ERR_NOT_FOUND -4
#define DECHLP_ERR_AMBIGUOUS -5
#define DECHLP_ERR_BUILD     -6
#define DECHLP_ERR_INVALID   -7
#define DECHLP_ERR_INTERNAL  -99

/* --- Match mode --- */

#define DECHLP_MATCH_ABBREVIATION 0
#define DECHLP_MATCH_EXACT        1

/* --- Opaque handle types --- */

typedef struct DecHlpLibrary DecHlpLibrary;
typedef struct DecHlpNavigator DecHlpNavigator;
typedef struct DecHlpBuilder DecHlpBuilder;

/* --- Error reporting --- */

/**
 * Return the last error message for the calling thread.
 * Returns a pointer to a null-terminated UTF-8 string.
 * The pointer is valid until the next FFI call on the same thread.
 * Returns NULL if no error has occurred.
 */
const char *dechlp_last_error(void);

/* --- Memory management --- */

/**
 * Free a string that was allocated by the FFI layer and returned to the caller.
 * Passing NULL is a safe no-op.
 */
void dechlp_string_free(char *s);

/* --- Library open/close --- */

/**
 * Open a compiled .hlib library file.
 *
 * On success, *out_lib is set to a valid handle and DECHLP_OK is returned.
 * On failure, *out_lib is set to NULL and a negative error code is returned.
 * The caller must close the handle with dechlp_library_close().
 *
 * @param path      Null-terminated path to the .hlib file.
 * @param out_lib   Receives the library handle on success.
 * @return          DECHLP_OK on success, negative error code on failure.
 */
int32_t dechlp_library_open(const char *path, DecHlpLibrary **out_lib);

/**
 * Close a library handle and release all associated resources.
 * Passing NULL is a safe no-op.
 * After this call, the handle is invalid and must not be used.
 */
void dechlp_library_close(DecHlpLibrary *lib);

/* --- Library header information --- */

/**
 * Get the number of nodes in the library.
 * Returns 0 if lib is NULL.
 */
uint32_t dechlp_library_node_count(const DecHlpLibrary *lib);

/**
 * Get the build timestamp (Unix epoch seconds) of the library.
 * Returns 0 if lib is NULL.
 */
uint64_t dechlp_library_build_timestamp(const DecHlpLibrary *lib);

/* --- Topic lookup --- */

/**
 * Look up a topic by a path of components.
 *
 * The path is specified as an array of null-terminated strings. For example,
 * to look up "COPY /CONFIRM", pass path = {"COPY", "/CONFIRM"} with
 * path_len = 2.
 *
 * On success (DECHLP_OK), the topic text is written to *out_text and its
 * byte length to *out_text_len. The text pointer borrows from the library's
 * memory map and is valid for the lifetime of the library handle. The text
 * is UTF-8 but NOT null-terminated; use out_text_len for the length.
 *
 * On DECHLP_ERR_NOT_FOUND or DECHLP_ERR_AMBIGUOUS, out_text and out_text_len
 * are not modified. Call dechlp_last_error() for details (including the list
 * of ambiguous candidates).
 *
 * @param lib           Library handle.
 * @param path          Array of path component strings.
 * @param path_len      Number of elements in path.
 * @param match_mode    DECHLP_MATCH_ABBREVIATION or DECHLP_MATCH_EXACT.
 * @param out_text      Receives pointer to topic body text (borrowed, not owned).
 * @param out_text_len  Receives byte length of topic body text.
 * @return              DECHLP_OK, DECHLP_ERR_NOT_FOUND, or DECHLP_ERR_AMBIGUOUS.
 */
int32_t dechlp_topic_lookup(
    const DecHlpLibrary *lib,
    const char *const *path,
    size_t path_len,
    int32_t match_mode,
    const char **out_text,
    size_t *out_text_len
);

/**
 * Get the display name of a topic at the given path.
 *
 * Returns the case-preserved topic name as stored in the library. The caller
 * must free the returned string with dechlp_string_free().
 *
 * @param lib           Library handle.
 * @param path          Array of path component strings.
 * @param path_len      Number of elements in path.
 * @param match_mode    DECHLP_MATCH_ABBREVIATION or DECHLP_MATCH_EXACT.
 * @param out_name      Receives a newly allocated null-terminated string (caller must free).
 * @return              DECHLP_OK, DECHLP_ERR_NOT_FOUND, or DECHLP_ERR_AMBIGUOUS.
 */
int32_t dechlp_topic_name(
    const DecHlpLibrary *lib,
    const char *const *path,
    size_t path_len,
    int32_t match_mode,
    char **out_name
);

/* --- Children enumeration --- */

/**
 * Count the number of direct children at a given topic path.
 *
 * Pass path = NULL and path_len = 0 to count root (level-1) topics.
 *
 * @param lib           Library handle.
 * @param path          Array of path component strings (NULL for root).
 * @param path_len      Number of elements in path (0 for root).
 * @param match_mode    DECHLP_MATCH_ABBREVIATION or DECHLP_MATCH_EXACT.
 * @param out_count     Receives the child count.
 * @return              DECHLP_OK or error code if path is invalid.
 */
int32_t dechlp_children_count(
    const DecHlpLibrary *lib,
    const char *const *path,
    size_t path_len,
    int32_t match_mode,
    size_t *out_count
);

/**
 * Get the name of the i-th child at a given topic path.
 *
 * Children are sorted alphabetically. Index is 0-based.
 * The caller must free the returned string with dechlp_string_free().
 *
 * @param lib           Library handle.
 * @param path          Array of path component strings (NULL for root).
 * @param path_len      Number of elements in path (0 for root).
 * @param match_mode    DECHLP_MATCH_ABBREVIATION or DECHLP_MATCH_EXACT.
 * @param index         0-based child index.
 * @param out_name      Receives a newly allocated null-terminated string.
 * @return              DECHLP_OK or DECHLP_ERR_INVALID if index is out of range.
 */
int32_t dechlp_children_name(
    const DecHlpLibrary *lib,
    const char *const *path,
    size_t path_len,
    int32_t match_mode,
    size_t index,
    char **out_name
);

/**
 * Get all child names at a given topic path as a single null-separated string.
 *
 * The returned string contains child names separated by null bytes, with a
 * double null terminator at the end. Example: "COPY\0DELETE\0RENAME\0\0".
 * The caller must free the returned string with dechlp_string_free().
 * *out_total_len receives the total byte length including all null bytes.
 *
 * This is more efficient than calling dechlp_children_name() in a loop.
 *
 * @param lib             Library handle.
 * @param path            Array of path component strings (NULL for root).
 * @param path_len        Number of elements in path (0 for root).
 * @param match_mode      DECHLP_MATCH_ABBREVIATION or DECHLP_MATCH_EXACT.
 * @param out_names       Receives the null-separated name string.
 * @param out_total_len   Receives total byte length of the name string.
 * @return                DECHLP_OK or error code.
 */
int32_t dechlp_children_names(
    const DecHlpLibrary *lib,
    const char *const *path,
    size_t path_len,
    int32_t match_mode,
    char **out_names,
    size_t *out_total_len
);

/* --- Build --- */

/**
 * Compile one or more .hlp source files into a .hlib binary library.
 *
 * @param input_paths   Array of null-terminated paths to .hlp source files.
 * @param input_count   Number of input files.
 * @param output_path   Null-terminated path for the output .hlib file.
 * @return              DECHLP_OK on success, DECHLP_ERR_BUILD or DECHLP_ERR_IO on failure.
 */
int32_t dechlp_build(
    const char *const *input_paths,
    size_t input_count,
    const char *output_path
);

/**
 * Compile with a progress callback.
 *
 * The callback is invoked for each topic processed, receiving the topic level
 * and name. The callback may be NULL for no progress reporting.
 *
 * @param input_paths   Array of input .hlp file paths.
 * @param input_count   Number of input files.
 * @param output_path   Output .hlib file path.
 * @param on_topic      Progress callback (may be NULL).
 * @param user_data     Opaque pointer passed through to the callback.
 * @return              DECHLP_OK on success, negative error code on failure.
 */
int32_t dechlp_build_with_callback(
    const char *const *input_paths,
    size_t input_count,
    const char *output_path,
    void (*on_topic)(uint8_t level, const char *name, void *user_data),
    void *user_data
);

/* --- Navigator --- */

/**
 * Create an interactive navigator for the given library.
 *
 * The library must remain open for the lifetime of the navigator.
 * The caller must destroy the navigator with dechlp_navigator_destroy().
 *
 * @param lib           Library handle (must outlive the navigator).
 * @param out_nav       Receives the navigator handle.
 * @return              DECHLP_OK on success.
 */
int32_t dechlp_navigator_create(
    const DecHlpLibrary *lib,
    DecHlpNavigator **out_nav
);

/**
 * Destroy a navigator and release its resources.
 * Passing NULL is a safe no-op.
 */
void dechlp_navigator_destroy(DecHlpNavigator *nav);

/**
 * Get the current depth of the navigator (0 = root).
 */
size_t dechlp_navigator_depth(const DecHlpNavigator *nav);

/**
 * Get the prompt string for the current navigator position.
 * The caller must free the returned string with dechlp_string_free().
 *
 * @param nav           Navigator handle.
 * @param out_prompt    Receives the prompt string.
 * @return              DECHLP_OK on success.
 */
int32_t dechlp_navigator_prompt(
    const DecHlpNavigator *nav,
    char **out_prompt
);

/**
 * Process a line of user input through the navigator.
 *
 * This is the main interactive entry point. The function processes the input
 * and returns a result code indicating what happened:
 *
 *   DECHLP_NAV_DISPLAY_TOPIC  - A topic was found. Call dechlp_navigator_topic_text()
 *                                and dechlp_navigator_topic_children() to get content.
 *   DECHLP_NAV_AMBIGUOUS      - Input was ambiguous. Call dechlp_last_error() for candidates.
 *   DECHLP_NAV_NOT_FOUND      - Input was not found. Call dechlp_last_error() for details.
 *   DECHLP_NAV_SHOW_TOPICS    - User typed "?". Call dechlp_navigator_topic_children().
 *   DECHLP_NAV_GO_UP          - Empty input at subtopic level. Navigator moved up.
 *   DECHLP_NAV_EXIT           - Empty input at root. Session should end.
 *
 * @param nav           Navigator handle.
 * @param input         Null-terminated user input string.
 * @param match_mode    DECHLP_MATCH_ABBREVIATION or DECHLP_MATCH_EXACT.
 * @return              One of the DECHLP_NAV_* result codes.

int32_t dechlp_navigator_input(
    DecHlpNavigator *nav,
    const char *input,
    int32_t match_mode
);

/* Navigator action result codes (positive, distinct from error codes) */
#define DECHLP_NAV_DISPLAY_TOPIC   1
#define DECHLP_NAV_AMBIGUOUS       2
#define DECHLP_NAV_NOT_FOUND       3
#define DECHLP_NAV_SHOW_TOPICS     4
#define DECHLP_NAV_GO_UP           5
#define DECHLP_NAV_EXIT            6
#define DECHLP_NAV_DISPLAY_MULTI   7

 *
 * The returned pointer borrows from the library's memory map and is valid
 * for the lifetime of the library handle.
 *
 * @param nav           Navigator handle.

    char **out_names,
```

### 2.5 Rust Implementation Structure

```rust
// dec-hlp-ffi/src/lib.rs

use std::cell::RefCell;
use std::ffi::{CStr, CString};
use std::os::raw::c_char;
use std::ptr;

// ---- Thread-local last-error ----

thread_local! {
    static LAST_ERROR: RefCell<Option<CString>> = RefCell::new(None);
}

fn set_last_error(msg: impl Into<String>) {
    let msg = msg.into();
    LAST_ERROR.with(|cell| {
        *cell.borrow_mut() = CString::new(msg).ok();
    });
}

fn clear_last_error() {
    LAST_ERROR.with(|cell| {
        *cell.borrow_mut() = None;
    });
}

#[unsafe(no_mangle)]
pub extern "C" fn dechlp_last_error() -> *const c_char {
    LAST_ERROR.with(|cell| {
        match cell.borrow().as_ref() {
            Some(cstr) => cstr.as_ptr(),
            None => ptr::null(),
        }
    })
}

// ---- Opaque handle types ----

// DecHlpLibrary wraps dec_hlp::library::Library.
// Boxed and cast to opaque pointer for C.
pub struct DecHlpLibrary {
    inner: dec_hlp::library::Library,
}

// DecHlpNavigator wraps dec_hlp::engine::Navigator.
// The navigator borrows from the library, so it holds a raw pointer
// to the library. The C API contract requires the library to outlive
// the navigator.
pub struct DecHlpNavigator {
    // Safety: the library pointer is valid for the lifetime of this struct
    // because the C API documents that the library must outlive the navigator.
    inner: dec_hlp::engine::Navigator<'static>,
    // Retains a reference to ensure the library is not closed prematurely.
    // In practice, the C caller is responsible for ordering; this field
    // is present for clarity.
    _library: *const DecHlpLibrary,
}

// ---- Panic guard ----

// All extern "C" functions use this wrapper to catch panics at the FFI
// boundary. A panic unwinding through extern "C" is undefined behavior.
fn catch_panic<F, T>(default: T, f: F) -> T
where
    F: FnOnce() -> T + std::panic::UnwindSafe,
{
    match std::panic::catch_unwind(f) {
        Ok(val) => val,
        Err(_) => {
            set_last_error("internal panic in dec-hlp FFI");
            default
        }
    }
}

// ---- Function implementations ----
// Each #[unsafe(no_mangle)] pub extern "C" fn validates inputs,
// calls into dec_hlp, and translates results to C conventions.
// Implementation follows the signatures in dec_hlp.h exactly.
```

### 2.6 Lifetime Management Across FFI

The central challenge is that `Navigator<'lib>` borrows from `Library`. In the FFI layer, lifetimes cannot be expressed. The solution:

1. `DecHlpLibrary` is heap-allocated (`Box<DecHlpLibrary>`) and returned as a raw pointer.
2. `DecHlpNavigator` holds a `Navigator<'static>` obtained by transmuting the lifetime. This is safe because the C API contract requires the library to outlive the navigator, and the library is pinned in heap memory.
3. The header documents this invariant clearly: "The library must remain open for the lifetime of the navigator."

This is a standard pattern used by `rusqlite`'s C bindings and similar projects.

### 2.7 Panic Safety

Every `extern "C"` function wraps its body in `std::panic::catch_unwind`. If a panic occurs:

1. The panic is caught and does not unwind across the FFI boundary (which would be UB).
2. `DECHLP_ERR_INTERNAL` (-99) is returned.
3. The panic message is stored in the thread-local last-error buffer.

### 3.3 Python API Design

The Python API is Pythonic, not a 1:1 mapping of the Rust API. It uses context managers, exceptions, iterators, and properties where appropriate.

```rust
// dec-hlp-python/src/lib.rs

use pyo3::prelude::*;
use pyo3::exceptions::{PyFileNotFoundError, PyValueError, PyRuntimeError};

// --- Custom exception types ---
// PyO3 supports creating Python exception classes from Rust.

pyo3::create_exception!(dec_hlp, DecHlpError, pyo3::exceptions::PyException);
pyo3::create_exception!(dec_hlp, LibraryError, DecHlpError);
pyo3::create_exception!(dec_hlp, FormatError, LibraryError);
pyo3::create_exception!(dec_hlp, TopicNotFoundError, DecHlpError);
pyo3::create_exception!(dec_hlp, AmbiguousTopicError, DecHlpError);
pyo3::create_exception!(dec_hlp, BuildError, DecHlpError);
pyo3::create_exception!(dec_hlp, ParseError, BuildError);

// --- HelpLibrary wrapper ---

#[pyclass]
struct HelpLibrary {
    // Option so we can implement close() and context manager __exit__
    inner: Option<dec_hlp::library::Library>,
    path: String,
}

#[pymethods]
impl HelpLibrary {
    #[new]
    fn new(path: &str) -> PyResult<Self> {
        let lib = dec_hlp::library::Library::open(std::path::Path::new(path))
            .map_err(|e| match e {
                dec_hlp::library::LibraryError::Io(ref io_err)
                    if io_err.kind() == std::io::ErrorKind::NotFound =>
                {
                    PyFileNotFoundError::new_err(format!("{path}: {e}"))
                }
                dec_hlp::library::LibraryError::InvalidFormat(_) => {
                    FormatError::new_err(e.to_string())
                }
                _ => LibraryError::new_err(e.to_string()),
            })?;
        Ok(HelpLibrary {
            inner: Some(lib),
            path: path.to_string(),
        })
    }

    fn __enter__(slf: PyRef<'_, Self>) -> PyRef<'_, Self> { slf }

    fn __exit__(
        &mut self,
        _exc_type: Option<&Bound<'_, PyAny>>,
        _exc_val: Option<&Bound<'_, PyAny>>,
        _exc_tb: Option<&Bound<'_, PyAny>>,
    ) -> bool {
        self.inner = None;
        false // Do not suppress exceptions
    }

    fn close(&mut self) {
        self.inner = None;
    }

    // ... property getters, lookup, topics, children, navigator ...
}

// --- Topic wrapper ---

#[pyclass]
struct Topic {
    name: String,
    level: u8,
    text: String,
    children: Vec<String>,
}

// Topic is a snapshot: data is copied out of the memory-mapped library
// at lookup time. This avoids lifetime issues (Python objects may outlive
// the library if the user is not careful with context managers).
// The cost is a string copy per lookup, which is negligible for help text.

// --- Navigator wrapper ---

#[pyclass]
struct Navigator {
    // Holds a Py<HelpLibrary> reference to prevent the library from being
    // garbage collected while the navigator is alive.
    library: Py<HelpLibrary>,
    // The actual navigator. Uses 'static lifetime via transmute,
    // safe because we prevent library deallocation via the Py reference.
    inner: Option<dec_hlp::engine::Navigator<'static>>,
}

// --- Module registration ---

#[pymodule]
fn _dec_hlp(m: &Bound<'_, PyModule>) -> PyResult<()> {
    m.add_class::<HelpLibrary>()?;
    m.add_class::<Topic>()?;
    m.add_class::<Navigator>()?;
    m.add_class::<NavResult>()?;
    m.add_class::<LibraryBuilder>()?;
    m.add_class::<BuildReport>()?;
    m.add("DecHlpError", m.py().get_type::<DecHlpError>())?;
    m.add("LibraryError", m.py().get_type::<LibraryError>())?;
    m.add("FormatError", m.py().get_type::<FormatError>())?;
    m.add("TopicNotFoundError", m.py().get_type::<TopicNotFoundError>())?;
    m.add("AmbiguousTopicError", m.py().get_type::<AmbiguousTopicError>())?;
    m.add("BuildError", m.py().get_type::<BuildError>())?;
    m.add("ParseError", m.py().get_type::<ParseError>())?;
    m.add_function(wrap_pyfunction!(build, m)?)?;
    m.add_function(wrap_pyfunction!(version, m)?)?;
    Ok(())
}
```

### 3.5 Data Copying Strategy

The Python bindings copy data out of the memory-mapped library rather than holding borrowed references. This is a deliberate tradeoff:

- **Safety.** Python's garbage collector may collect the `HelpLibrary` while a `Topic` object is still alive. By copying, `Topic` objects are self-contained and always valid.
- **Cost.** Help text bodies are typically a few hundred bytes. Copying is negligible compared to Python object creation overhead.
- **Simplicity.** Avoids complex `Py<HelpLibrary>` reference tracking in every `Topic`. The `Navigator` does hold a `Py<HelpLibrary>` reference because it is explicitly tied to a library instance with a longer-lived interaction pattern.

## 6. Design Decisions Summary

### Why opaque handles rather than exposing Rust struct layouts?

Rust struct layouts are not stable across compiler versions and are not guaranteed to match C expectations. Opaque handles (`typedef struct Foo Foo;`) hide the layout entirely, requiring all access through function calls. This is the standard approach used by SQLite, OpenSSL, and essentially every mature C library.

### Why thread-local last-error rather than out-parameters?

Out-parameter error strings require the caller to manage string buffers or the library to allocate strings on every call. The thread-local pattern is simpler: the error is stored once and retrieved optionally. The only pitfall (error from one call clobbered by another) is mitigated by checking immediately after the failing call. This matches the `errno` convention that C programmers already understand.

### Why copy data in the Python bindings?

Python objects have unpredictable lifetimes controlled by the garbage collector. Holding borrowed references into memory-mapped files would require preventing library deallocation while any Topic exists. This is possible (via `Py<HelpLibrary>` references) but adds complexity and fragility. Copying a few hundred bytes of help text per lookup is essentially free compared to Python object overhead.

### Why not expose `LibrarySet` in the FFI?

The core library's `LibrarySet` merges multiple libraries at query time. For the C API, this is deferred to a future version. The initial API covers single-library access, which is the common case. Multi-library support can be added later with `dechlp_library_set_*` functions without breaking the existing API. For Python, users can open multiple `HelpLibrary` instances and query them individually.

### Why `maturin` rather than `setuptools-rust`?

Maturin is purpose-built for PyO3 projects, handles cross-compilation, produces manylinux-compatible wheels, and integrates with the `PyO3/maturin-action` GitHub Action. It requires less configuration than `setuptools-rust` and is the recommended build tool in the PyO3 documentation.

### Why not C++ bindings?

The C header is directly usable from C++ (via the `extern "C"` block and `#ifdef __cplusplus` guards). A C++ wrapper with RAII classes (e.g., `std::unique_ptr<DecHlpLibrary, decltype(&dechlp_library_close)>`) can be written as a header-only layer by consumers. Shipping a C++ wrapper would force a specific C++ standard version and style; the C API is the universal foundation.
